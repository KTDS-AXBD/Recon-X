# Design: TD-41 CF Access JWT mock E2E 복원

## 1. CF_Authorization JWT 페이로드 스키마

CF Access가 발급하는 JWT는 다음 클레임을 포함한다 (`src/lib/auth.ts:CfJwtClaims` 기준):

```typescript
interface CfJwtClaims {
  email: string;          // "e2e@test.local"
  name?: string;          // "E2E Test User"
  sub: string;            // "e2e-sub-001"
  aud: string | string[]; // ["e2e-audience"]  — CF Access Application ID
  iss: string;            // "https://e2e.cloudflareaccess.com"
  iat: number;            // Math.floor(Date.now() / 1000)
  exp: number;            // iat + 3600 (1시간)
}
```

### JWT 빌더 설계 (`cf-access-cookie.ts`)

```typescript
// 서명 없이 base64url encode만 수행 (클라이언트 decode는 검증 없이 payload만 파싱)
// getCfJwtFromCookie()가 signature 검증을 하지 않으므로 안전
export function buildCfJwt(overrides?: Partial<CfJwtClaims>): string
export function buildCfAuthorizationCookie(): { name, value, domain, path }
```

JWT 구조: `header.payload.` (빈 signature) — Playwright 단독 환경에서 CF edge 검증 없음.
클라이언트 `parseCfJwt()`는 만료 시간만 검사하므로 `exp = now + 3600`으로 설정.

## 2. /auth/me 응답 스키마 (`auth-me-response.ts`)

```typescript
export const AUTH_ME_STUB = {
  email: "e2e@test.local",
  role: "engineer",          // CfUser["role"]
  status: "active",          // CfUser["status"]
} as const;
```

`page.route('**/auth/me', route => route.fulfill({ status: 200, contentType: 'application/json', body: JSON.stringify(AUTH_ME_STUB) }))` 패턴으로 사용.

## 3. Cookie 도메인/Path 설정

```typescript
{
  name: "CF_Authorization",
  value: buildCfJwt(),
  domain: "localhost",
  path: "/",
  httpOnly: false,    // document.cookie 접근 필요 (AuthContext)
  secure: false,      // HTTP localhost
  sameSite: "Lax",
}
```

Playwright `addCookies()` 호출 위치: `auth.setup.ts` — page.context()에 직접 추가 후 storageState 저장.

## 4. Worker 파일 매핑

| 파일 | 변경 유형 | 내용 |
|------|-----------|------|

## 5. 통합 검증 테스트 설계 (`auth.spec.ts`)

추가할 테스트:
```typescript
test("CF JWT mock: auth/me stub returns engineer role", async ({ browser }) => {
  // Fresh context (no storageState) — VITE_DEMO_MODE=1 에서도 CF JWT path 테스트 가능한 방법:
  // 실제로는 VITE_DEMO_MODE=1이면 demo bypass가 CF JWT보다 우선됨.
  // 이 테스트는 구조 검증용: cookie + route mock이 올바르게 설정되는지 확인.
  const ctx = await browser.newContext({ storageState: undefined });
  const page = await ctx.newPage();
  
  // CF_Authorization cookie 주입
  await ctx.addCookies([buildCfAuthorizationCookie()]);
  
  // /auth/me stub 설정
  await page.route('**/auth/me', route => route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify(AUTH_ME_STUB),
  }));
  
  // 검증: cookie가 올바르게 추가됨
  const cookies = await ctx.cookies();
  const cfCookie = cookies.find(c => c.name === 'CF_Authorization');
  expect(cfCookie).toBeDefined();
  expect(cfCookie?.value).toContain('.');  // JWT format check
  
  await ctx.close();
});
```

이 테스트는 VITE_DEMO_MODE=1 환경에서도 infrastructure 유효성만 검증 (CF JWT 경로 실 activation은 별도 환경 필요).

## 6. auth.setup.ts 변경 설계

```typescript
// 기존 demo bypass 유지 + CF_Authorization cookie 추가
setup("authenticate ...", async ({ page }) => {
  // 1. CF_Authorization cookie 주입 (storageState에 포함)
  await page.context().addCookies([buildCfAuthorizationCookie()]);
  
  // 2. 기존 demo bypass (VITE_DEMO_MODE=1 환경)
  await page.goto("/?demo=1");
  await page.waitForURL(/\/executive\/overview/, { timeout: 15_000 });
  
  // 3. storageState 저장 (cookie + localStorage 포함)
  await page.context().storageState({ path: AUTH_FILE });
});
```

결과: `e2e/.auth/user.json`에 `CF_Authorization` cookie가 포함되어 모든 spec에 자동 전파.

## 7. 리스크

| 리스크 | 완화 |
|--------|------|
| JWT `exp` 만료 (1시간 제한) | auth.setup.ts 실행 시점에 동적 생성 — CI 1h 초과 없음 |
| VITE_DEMO_MODE=1에서 CF JWT 경로 미활성화 | 테스트는 infrastructure 검증만 (경로 activation은 향후 VITE_DEMO_MODE 제거 시) |
| storageState에 cookie domain 불일치 | domain=localhost, Playwright webServer baseURL=http://localhost:5173 → 일치 |
