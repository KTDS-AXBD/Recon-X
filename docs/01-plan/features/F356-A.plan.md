## 1. Overview

### 1.1 Purpose

Phase 2 본개발 결과물의 **AI-Ready 품질**을 6기준으로 자동 평가할 수 있는 LLM 기반 채점기를 PoC 단계로 구현한다. Phase 1은 **스키마/프롬프트/비용 가드 확정 + 80건 샘플 정확도 실측**에 집중하고, Phase 2 (F356-B, Sprint 231~232)에서 전수 5,214 스케일 + API 엔드포인트로 확장한다.

### 1.2 Background

- **상위 계획**: `docs/01-plan/features/phase-3-gap-remediation.plan.md` §G-3 Phase 1 (AIF-PLAN-037)
- **PRD**: `docs/req-interview/decode-x-v1.3-phase-3/prd-final.md` §C.1 (6기준 확정)
- **Phase 2 데이터 기반**: Sprint 228 G-1 Phase 3에서 Tier-A 7 lpon-* container packaging 완료(AI-Ready mean 0.916 간단 집계). 본 작업은 이 단순 집계를 **6기준 독립 LLM 평가로 대체**
- **착수 근거**: G-1 Phase 3 MERGED로 Tier-A skill 데이터 확보 → G-3 PoC 의존성 해소. Plan-037 Sprint 229~230 예정이었으나 Sprint 229는 F383 (AXIS DS Tier 3)로 선점 → Sprint 230 단독 착수

### 1.3 6기준 (PRD §C.1 확정)

1. **소스코드 정합성** — Java 소스 vs spec-container YAML 필드 일치율
2. **주석·문서 일치** — Javadoc/inline comment가 rules/runbook 내용과 일치하는가
3. **입출력 구조 명확성** — API 스키마 + DTO 정의 vs 실 구현 I/O 경계 명확성
4. **예외·에러 핸들링** — throw/catch 패턴 + error code 체계성
5. **업무루틴 분리·재사용성** — SRP 준수 + public method 재사용 설계
6. **테스트 가능성 및 단위테스트 적합성** — 외부 의존성 mock 용이성 + 순수 함수 비율

## 3. Execution Plan — 4 Step × 2h

### Step 1. 스키마 설계 (2h)

**파일**: `packages/types/src/ai-ready.ts`

**산출물**:
- `AIReadyCriterion` enum: `source_consistency | comment_doc_alignment | io_structure | exception_handling | srp_reusability | testability`
- `AIReadyScore` Zod schema: `{ criterion, score: 0~1, rationale: string, passThreshold: 0.75 }`
- `AIReadyEvaluation` Zod schema: `{ skillId, criteria: AIReadyScore[], totalScore: number, passCount: number, evaluatedAt, modelVersion }`
- 총점 산식: **단순 평균** (Sinclair 확정, 가중 평균은 Phase 2에서 데이터 보고 재결정)
- `passThreshold: 0.75` — 각 기준 0.75 이상 통과, 6기준 중 4기준 이상 통과 시 AI-Ready PASS

**검증**: 타입 export + `vitest` 스키마 validation 테스트 5종

### Step 2. LLM 프롬프트 (2h)

**파일**: `services/svc-skill/src/ai-ready/prompts.ts`

**산출물**:
- `buildPrompt(criterion, skillSourceCode, skillMetadata): string` — 6기준 각 독립 프롬프트
- 입력 구조: Java 소스 코드 (rules/runbooks 참조) + YAML/JSON metadata (provenance.yaml + contracts)
- 출력 구조: `{ score: number, rationale: string }` JSON 강제 (LLM JSON mode 사용)
- 프롬프트 원칙: 각 기준당 **150~250 단어**, 예시 포함, `rationale`에 source line 인용 유도

**검증**: 10 skill 대상 dry-run → JSON 파싱 성공률 100%, score 분포 sanity check (0/1 극단 쏠림 없음)

### Step 3. PoC 스크립트 (2h)

**파일**: `scripts/ai-ready/evaluate.ts`

**산출물**:
- CLI: `pnpm tsx scripts/ai-ready/evaluate.ts --sample 80 --model haiku`
- 샘플링: **Tier-A 집중 40건** (lpon-charge/budget/gift/payment/purchase/refund/settlement 분배) + **무작위 40건** (pension + 온누리 기타)
- LLM 호출: `callLlm()` from `@ai-foundry/utils/llm-client` — svc-llm-router HTTP REST, tier=haiku 기본 (Opus는 10건만 교차 검증)
- **비용 가드**:
  - 실행 전 `GET /llm-router/usage?date={today}` 호출하여 누적 비용 확인
  - $25 초과 시 경고 출력 후 계속
  - $30 초과 시 즉시 중단 + AskUserQuestion (계속/중단/샘플 축소)
  - 호출마다 예상 비용 누적 log
- 출력: `reports/ai-ready-poc-{date}.json` — 80 × 6 = 480 점수 raw data + 전체 집계

**검증**: 비용 가드 unit test (mock usage endpoint) + 80건 실행 total cost 제한 내(~$15 예상, Haiku 기준)

### Step 4. 수기 검증 (2h)

**산출물**:
- 10% 샘플(8건) 수기 재채점 — Tier-A 4건 + 무작위 4건 분배
- 정확도 지표: LLM score vs 수기 score **각 기준당 |diff| ≤ 0.1** 이내 비율
- 리포트: `reports/ai-ready-poc-accuracy-{date}.md`
  - 8건 × 6기준 = 48 pair 비교표
  - 전체 정확도 % + 기준별 정확도 분해
  - 실패 케이스 원인 분석 (프롬프트 개선 TODO)

**판정**:
- ✅ **정확도 ≥ 80%** (48 pair 중 ≥ 38 pair 일치) → Phase 2 (F356-B) 승격 GO
- ⚠️ **정확도 60~80%** → 프롬프트 개선 iterate 1회 (동일 8건 재평가)
- ❌ **정확도 < 60%** → 스키마/프롬프트 근본 재설계 (Sprint 연장 또는 scope 축소)
