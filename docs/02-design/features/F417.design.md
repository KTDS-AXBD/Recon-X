# F417 Design — LPON skill source data 고도화 PoC

## 1. 배경 및 Root Cause

### 현재 `spec-content-adapter.ts` 문제점

```typescript
// buildRulesTable() — exception 컬럼 (line 75)
const exception = p.source.excerpt ?? "—";
// LPON 번들: source.excerpt = 원문 발췌 텍스트 (예: "퇴직연금 중도인출 규정 제15조")
// → exception/else 비즈니스 룰과 무관 → exception_handling rubric 0점

// tests 생성 (lines 38-39)
`  - "${p.condition} 조건 충족"`,   // ← 추상적 → io_structure / testability 0점
```

### Rubric 실패 조건

| Criterion | 실패 원인 | Fix Target |
|-----------|---------|-----------|
| exception_handling | rules/*.md exception 컬럼 = "—" 또는 무관 텍스트 | `source.excerpt` → exception clause |
| io_structure | tests given/then에 구체적 필드값/타입 없음 | test scenarios에 concrete 필드 삽입 |
| testability | happy path만 1건, error case 0건 | 3시나리오(happy/edge/error) 생성 |

## 2. 설계 결정 (Design Decisions)

### DD-1: Augmented Bundle 저장 경로
- `skill-packages/augmented/{skillId}.skill.json` (신규 R2 prefix)
- 원본 번들 보존 (원본 `r2_key`는 D1에서 변경 안 함)

### DD-2: Augmented Data 인코딩 방식
- **exception clause** → `policy.source.excerpt` 필드 교체 (이미 adapter가 exception 컬럼으로 사용)
- **test scenarios** → `policy.description` 필드에 `---TEST_SCENARIOS---` 마커 + YAML 삽입
  - `PolicySchema.description: z.string().optional()` 이미 존재하므로 schema 변경 없음

### DD-3: Augmented R2 Key 전달 방식
- `POST /skills/{id}/ai-ready/evaluate` 요청 바디에 `r2KeyOverride?: string` 추가
- D1 마이그레이션 불필요, 한시적 PoC 경계 명확
- `X-Internal-Secret` 헤더 필수 (internal-only 기능)

### DD-4: Test Scenarios 파싱 방식
- `spec-content-adapter.ts:tests` 생성 시 `p.description`이 `---TEST_SCENARIOS---\n` 로 시작하면 파싱
- 파싱 실패 또는 마커 없으면 기존 템플릿 fallback

## 3. 데이터 흐름

```
[augment-skill-data.ts]
  1. R2에서 번들 다운로드 (r2_key from D1 또는 직접 CF R2 API)
  2. 각 policy에 대해 Haiku 2-prompt 병렬 호출:
     a. Exception Prompt → source.excerpt 교체
     b. Test Scenarios Prompt → description 삽입
  3. 42건 augmented SkillPackage → R2 write
     (skill-packages/augmented/{skillId}.skill.json)
  4. /tmp/augmented-ids.txt 생성 (재평가 script용)

[augment-eval-loop.ts]
  1. /tmp/augmented-ids.txt 읽기
  2. POST /skills/{id}/ai-ready/evaluate
     body: { force: true, model: "haiku", r2KeyOverride: "skill-packages/augmented/{id}.skill.json" }
  3. D1에 저장 + 결과 리포트

[svc-skill evaluate route — MODIFIED]
  1. r2KeyOverride 바디 필드 수신
  2. loadSpecContent(env, skillId, orgId, r2KeyOverride ?? skillRow.r2_key)

[spec-content-adapter.ts — MODIFIED]
  tests 생성 시:
  - description이 ---TEST_SCENARIOS--- 마커 포함 → YAML 파싱 → 구체 시나리오 사용
  - 없으면 기존 추상 템플릿 사용
```

## 4. LLM Augmentation Prompts

### Prompt A: Exception Clause

```
다음 퇴직연금/온누리상품권 업무 policy rule에서 exception(Else) 조건을 추론하라.
예외 조건이란 위 policy가 적용되지 않거나 거부되는 상황이다.

Policy:
- Code: {code}
- Title: {title}
- Condition (When): {condition}
- Criteria (If): {criteria}
- Outcome (Then): {outcome}

출력 형식: 한 문장 한국어. 예시: "예외: 신청 금액이 0원 이하이거나 필수 증빙서류 미제출 시 처리 거부."
예외가 명확하지 않으면: "예외: 해당 없음 (조건 충족 시 항상 처리)."
출력: 예외 문장만, 다른 텍스트 없이.
```

### Prompt B: Test Scenarios

```
다음 업무 policy rule에 대한 테스트 시나리오 3개를 YAML로 생성하라.

Policy:
- Code: {code}
- Title: {title}
- Condition (When): {condition}
- Criteria (If): {criteria}
- Outcome (Then): {outcome}
- Exception (Else): {exception_clause}

규칙:
1. happy_path: 정상 처리 (구체적 입력값과 예상 결과)
2. edge_case: 경계값 또는 최소/최대 조건
3. error_case: 예외 조건 발동 (error_code 포함)
4. given 필드: 구체적 변수명과 값 사용 (boolean 제외, 숫자/문자열/날짜)

출력 형식:
---TEST_SCENARIOS---
- id: happy_path
  given:
    {변수명}: {구체적값}
  when:
    action: {처리 액션명}
  then:
    outcome: {결과}
    result_code: {SUCCESS_CODE}
- id: edge_case
  given:
    {변수명}: {경계값}
  when:
    action: {처리 액션명}
  then:
    outcome: {결과}
    result_code: {CODE}
- id: error_case
  given:
    {변수명}: {오류 유발값}
  when:
    action: {처리 액션명}
  then:
    outcome: rejected
    error_code: {ERR_CODE}

출력: YAML만, 다른 텍스트 없이.
```

## 6. R2 접근 방법

`augment-skill-data.ts`에서 R2 번들을 읽는 방법:
- **Option A** (Adopted): CF R2 REST API `GET https://api.cloudflare.com/client/v4/accounts/{id}/r2/buckets/{bucket}/objects/{key}` — 외부 스크립트에서 직접 접근 가능
- Option B: svc-skill `/skills/{id}` API 경유 → r2_key 추출 후 GET — API 호출 2단계 필요

CF R2 API 필요 환경변수: `CF_ACCOUNT_ID`, `CF_R2_API_TOKEN`, `R2_BUCKET_NAME`

## 7. 비용 추정

| 항목 | 건수 | 단가 | 예상 비용 |
|-----|-----|------|---------|
| Haiku exception prompt | 42 skills × ~5 policies avg = 210 calls | ~$0.003/call | ~$0.63 |
| Haiku test scenarios prompt | 210 calls | ~$0.004/call | ~$0.84 |
| 재평가 (6 criteria × 42 skills) | 252 calls | ~$0.0012/call | ~$0.30 |
| **합계** | | | **~$1.77** (≤ $1.80 목표) |

> Haiku input 단가: $0.80/M tokens. 평균 prompt ~500 tokens = $0.0004/call. 출력 ~100 tokens = $0.0004/call. 총 ~$0.0008/call. 위 추정은 여유분 포함.

## 9. spec-content-adapter.ts 수정 설계

```typescript
// tests 생성 부분 수정
const tests = pkg.policies.map((p) => {
  // DD-4: description에 ---TEST_SCENARIOS--- 마커가 있으면 파싱해서 사용
  if (p.description?.startsWith("---TEST_SCENARIOS---")) {
```

## 10. 실패 기준

| 조건 | 처리 |
