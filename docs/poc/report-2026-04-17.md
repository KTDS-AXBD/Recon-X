## 3. 6기준 설계 (핵심만)

**원칙**: 규칙 기반 deterministic 채점 — LLM 없음. 결정적/재현성/빠름/공짜.

| # | 기준 | 주요 Signal | 임계값 |
|:-:|------|-------------|:------:|
| 1 | Machine-readable | `$schema` 존재 · Zod 검증 · Policy code regex | 0.9 |
| 2 | Semantic Consistency | `termUris` · `skosConceptScheme` · 단일 도메인 | 0.7 |
| 3 | Testable | policy 본문 ≥20자 · `excerpt` 존재 · 개수 ≥3 | 0.7 |
| 4 | Traceable | `sourceDocumentIds` · stage 체인 ≥3 · 정책↔문서 매핑 | 0.8 |
| 5 | **Completeness (B+T+Q)** | (세부는 아래 섹션 4) | 0.67 |
| 6 | Human-reviewable | `trust.level=reviewed` · title 길이 · author · 본문 길이 | 0.6 |

**Overall = 6기준 평균**, ≥ 0.8이면 "AI-Ready"

## 4. 기준 5 Completeness — 재설계 (PoC 중 수정됨)

**초기 설계**: `adapters.mcp/openapi` 필드 유무로 Technical 판정
**문제 발견**: 해당 필드가 관행적으로 미사용 → 모든 skill이 T=0 판정 → 채점기가 **구조적으로 무의미**
**재설계 (경험치 기반)**: 정책 텍스트에서 signal 추출

| 차원 | Signal 3종 | 설명 |
|------|-----------|------|
| **Business** | 규칙 완성도 + 도메인 키워드 매칭 + 수치 존재 | SKILL_CATEGORIES 16카테고리 94키워드 재활용 |
| **Technical** | API/엔드포인트 패턴 + 필드/스키마 키워드 + 식별자 regex (+ adapters bonus) | policy 본문/excerpt에서 추출 |
| **Quality** | 비기능 키워드 21종 + trust.score + excerpt 보유 비율 | SLA·보안·성능 관점 |

이 재설계가 **본 PoC의 핵심 통찰**을 만들었다 — "B는 잘 뽑는데 T는 거의 안 뽑는다"가 정량화됐다.

## 5. 결과 해석 (Why It Matters)

### 잘 되고 있는 것 (ignore-able)
- ✅ **구조적 품질 (기준 1, 2, 4, 6)**: 90%+ — Cloudflare 파이프라인, Zod 검증, Neo4j upsert, HITL 검토 모두 안정화
- ✅ Business 규칙 추출 (91.8%) — condition/criteria/outcome 구조, 도메인 키워드, 수치 — LLM 프롬프트가 업무 규칙은 잘 뽑음

### 해결해야 할 것 (what to fix)

#### 🎯 #1 우선: Technical 관점 추출 (현 4.3% → 목표 60%+)
- **원인 추정**: svc-extraction 프롬프트가 "정책/업무 규칙"에 집중, API·필드·스키마를 policy 본문에 포함 안 함
- **해결**: prompts/policy.ts에 Technical 지시 추가 — "해당 정책이 적용되는 API endpoint·주요 필드·데이터 구조를 함께 명시하라"
- **예상 효과**: 약 500건(56%) 추가 AI-Ready 전환 가능

#### 🎯 #2: Quality 관점 강화 (현 27.0% → 목표 60%+)
- excerpt 필수화 + 비기능 키워드 추출 프롬프트 분기
- 원본 문서에서 성능/보안/SLA 관련 섹션 명시적 추출

#### #3: Testable 개선 (현 38.5% → 70%+)
- policy 본문 최소 길이 20자 검증 + 미달 시 재추출
- `source.excerpt` optional → 필수 전환

### Bottom 20% 원인 다름
- Traceable=0 + Semantic 0.30 — **Ontology/Provenance 결손** — Stage 4의 graceful fallback이 너무 허용적
- 이건 Deep Dive 구현과 별개 — 기존 파이프라인 디버깅으로 해결
