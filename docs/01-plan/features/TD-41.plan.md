# Plan: TD-41 CF Access JWT mock E2E 복원

## 1. 배경

Sprint 227 F401에서 `VITE_DEMO_MODE=1` + `?demo=1` 우회로 E2E 45/45 CI PASS를 달성했다.
그러나 이 접근은 빌드 타임 플래그로 AuthContext 코드 경로를 분기해 실 CF Access JWT 흐름을 검증하지 않는다.

Sprint 253 목표: 정공 CF Access JWT mock 인프라 도입
- `CF_Authorization` cookie를 Playwright가 직접 주입
- `/auth/me` API를 `page.route()` stub으로 처리
- 기존 45/45 PASS 유지 + CF JWT 경로 1회 통합 검증

## 2. 현재 Auth 흐름

```
CI (VITE_DEMO_MODE=1)
  auth.setup.ts → page.goto("/?demo=1") → localStorage.__demo_user__
  → storageState(e2e/.auth/user.json) 저장
  → 모든 spec: localStorage에서 demo user 직접 로드 (CF JWT 미경유)
```

```
Production
  브라우저 → CF Access → CF_Authorization cookie 발급
  → AuthContext.getCfJwtFromCookie() → /auth/me API → role 조회
```

## 3. msw vs page.route 비교

| 항목 | msw (Service Worker) | page.route() (Playwright native) |
|------|---------------------|----------------------------------|
| 설정 복잡도 | 높음 (SW 등록 필요) | 낮음 (1줄) |
| Playwright 호환 | Service Worker 라이프사이클 충돌 가능 | 완전 호환 |
| storageState 저장 | 불가 (SW는 persistence 별도) | 불가 (route는 context 한정) |
| 타입 지원 | @types/msw 필요 | Playwright 내장 |

**결론**: `page.route()` 전용으로 결정. msw는 이점 없이 SW 충돌 위험만 있음.

## 4. globalSetup 영향 분석

`auth.setup.ts` (Playwright `setup` project)에서 설정한 `page.route()`는 **storageState에 저장되지 않는다**.
따라서 개별 spec에서 `/auth/me` stub이 필요할 경우 각자 `page.route()`를 호출해야 한다.

**결정**: 
- `auth.setup.ts`: CF_Authorization cookie를 context에 추가 (storageState에 cookie 포함)
- `e2e/fixtures/auth/`: 공유 fixture 모듈 신설 (개별 spec에서 import하여 사용)
- globalSetup 방식은 도입하지 않음 (route 저장 불가로 이점 없음)
