### Type A: 테이블형 데이터 (목록/정의서/추적표)

**해당 문서**: 요구사항정의서, Gap분석서, 인터페이스목록, 화면목록, 배치JOB목록, 코드정의서, 요구사항추적표, 통합테스트시나리오

**구조 특징**:
- 표지 시트 + 제개정이력 시트 + 본문 시트 + 샘플 시트 + 작성가이드 시트
- 본문 시트는 **정형 테이블** (header row + data rows)
- 셀 병합(merges) 적음 (0~20개)
- 행 수: 수십~수천 행 (Gap분석서 1,621행, 코드정의서_상세 43,997행)

**대표 컬럼 패턴**:
```
요구사항정의서: No | Level1 | Level2 | Level3 | 요구사항ID | 요구사항명 | 상세정의 | 전제조건
Gap분석서:     GAP ID | Level1 | Level2 | Level3 | 기능ID | 기능명 | GAP상세정의
인터페이스목록: No | 대분류 | 중분류 | 소분류 | 인터페이스ID | 인터페이스명 | 설명 | 송수신구분
화면목록:      No | 대분류 | 중분류 | 소분류 | 화면ID | 화면명 | 설명 | 유형 | 개발유형
배치JOB목록:   No | 대분류 | 중분류 | 소분류 | BatchJobID | Application명 | 설명 | 수행주기
```

**개선 포인트**:

### Type B: 폼 기반 설계서 (화면설계서)

**해당 문서**: 450건 (07.화면설계서 371 + 06.화면설계서 79)

**구조 특징**:
- 101×78 등 **넓은 격자** (78컬럼은 엑셀에서 픽셀 단위 레이아웃 사용)
- **대량 셀 병합** (239~279개 merges per sheet)
- 데이터가 **폼 필드/값 쌍**으로 배치 (테이블이 아님)
- 각 시트 = 하나의 화면 탭 (신청 Tab, 이력조회 Tab 등)

**데이터 구조** (화면설계서 공통 레이아웃):
```
R1:  시스템명 (퇴직연금시스템)
R3:  화면명 | [값] | 화면ID
R4:  대분류 | [값] | 중분류 | [값]
R5:  화면설명 | [값]
R6:  서비스클래스ID | [값]
R7:  "1. 매뉴 레이아웃"
R8~R40: UI 필드 레이아웃 (label-value 쌍, 셀 병합으로 구현)
R46: "2. (참고) 기존 시스템 레이아웃"
R47+: 3.데이터 구성항목 (항목명 | 컨트롤유형 | 필수 | I/O/H | 타입)
R58+: 4.처리로직 (이벤트명 | 파라미터 | 처리내용)
R66+: 5.현업 추가 설명
```

**현재 parseXlsx 적합도**: ⚠️ 낮음
- 78컬럼 Markdown 테이블 → 가독성 극히 낮음
- 셀 병합 정보 무시 → label-value 매핑 불가
- UI 레이아웃 영역과 데이터 정의 영역 구분 불가

**개선 전략** (핵심):
1. **메타데이터 추출**: R1~R6의 화면명/화면ID/대분류/중분류/서비스클래스ID를 구조화
2. **섹션 감지**: "1.매뉴 레이아웃" / "3.데이터 구성항목" / "4.처리로직" / "5.현업 설명" 마커로 분할
3. **데이터 구성항목 → 정형 테이블**: 항목명, 컨트롤유형, 필수, I/O/H, 타입
4. **처리로직 → 정형 테이블**: 이벤트명, 파라미터, 처리내용
5. **UI 레이아웃 → key-value 추출**: 셀 병합 범위를 활용하여 label-value 쌍 감지

## 4. 파싱 전략 권장

### 우선순위 1: 화면설계서 전용 파서 (450건, 전체의 58%)

**문제**: 현재 parseXlsx는 78컬럼 폼을 Markdown 테이블로 변환 → Stage 2에서 의미 추출 불가

**권장 파서 설계**:
```
parseScreenDesign(fileBytes, fileName):
  1. 메타데이터 추출 (화면명, 화면ID, 서비스클래스ID, 대분류/중분류)
  2. 섹션 분할 ("1.레이아웃" / "3.데이터구성" / "4.처리로직" / "5.현업설명")
  3. 데이터구성항목 → 정형 테이블 (항목명, 컨트롤유형, 필수, I/O/H, 타입)
  4. 처리로직 → 정형 테이블 (이벤트명, 파라미터, 처리내용)
  5. UI 필드 → key-value 쌍 (셀 병합 범위 활용)
  → UnstructuredElement[] (XlScreen:<siSubtype> 타입)
```

### 우선순위 2: 표지/샘플/가이드 시트 스킵

**문제**: 모든 문서에 표지, 제개정이력, 샘플, 작성가이드 시트가 포함 → 노이즈

**권장**: 시트명 기반 필터링
```
SKIP_SHEETS = ["표지", "제개정이력", /샘플/, /작성가이드/]
```

### 우선순위 3: 프로그램설계서 메타데이터 추출 강화

**현재**: R1~R5 메타데이터가 데이터 행으로 처리됨
**권장**: R3~R4의 프로그램ID/프로그램명/담당자를 메타데이터로 분리, R6 이후를 테이블로 처리

### 우선순위 4: SCDSA002 파일 대응

4건의 암호화 파일(테이블설계서 3 + 메뉴구조도 1)은 도메인 ERD 이해에 핵심.
- 옵션 A: 원본(비암호화) 파일 확보
- 옵션 B: SCDSA002 복호화 라이브러리 조사
- 옵션 C: 해당 정보를 다른 소스(개발가이드 등)에서 부분 추출

## 5. 파이프라인 영향 분석

### Stage 1 (Ingestion) — 직접 영향
- **화면설계서 전용 파서** 추가 필요
- 시트 스킵 로직 추가
- SiSubtype에 신규 유형 추가 불필요 (기존 11개로 커버)

### Stage 2 (Extraction) — 간접 영향
- 화면설계서의 "처리로직" 섹션이 프로세스/규칙 추출의 핵심 입력
- 프로그램설계서의 "처리내용" 컬럼이 비즈니스 로직 기술
- 파서 개선으로 LLM에 전달되는 텍스트 품질이 크게 향상될 전망

### Stage 3 (Policy Inference) — 간접 영향
- 화면설계서 "처리로직" + 프로그램설계서 "처리내용"에서 조건-기준-결과 트리플 추출 가능성 높음
- 450건 화면설계서 × 평균 2시트 × 처리로직 = ~900개 이벤트 처리 규칙 예상
