# F408 Design — TD-47 evaluator R2 경로 변경

## 1. 배경 및 목표

**문제**: `evaluator.loadSpecContent`가 R2 `spec-containers/{org}/{skillId}/manifest.json` 레이아웃을 기대하나
production R2는 `skill-packages/{skillId}.skill.json` 단일 번들(859건)만 존재. 7건만 커버 가능.

**목표**: evaluator를 `skill-packages/{skillId}.skill.json`에서 직접 SpecContent를 추출하도록 수정.
기존 spec-containers 경로는 `loadSpecContentLegacy`로 보존(Tier-A 7건 디버깅용).

## 2. 설계 결정 (사용자 승인 완료)

| 결정 | 내용 |
|------|------|
| 옵션 B 채택 | evaluator 경로를 `skill-packages/{id}.skill.json`으로 변경 |
| Legacy 보존 | 기존 `loadSpecContent` → `loadSpecContentLegacy`로 rename, export 유지 |
| Adapter 분리 | `spec-content-adapter.ts` 신설 — SkillPackage → SpecContent 변환 책임 |
| organizationId 유지 | 함수 시그니처 호환 — caller 수정 없음 |

## 3. 파일 매핑 (Worker 1개)

| 파일 | 변경 유형 | 내용 |
|------|-----------|------|
| `services/svc-skill/src/ai-ready/spec-content-adapter.ts` | **신설** | SkillPackage → SpecContent 변환 |
| `services/svc-skill/src/ai-ready/evaluator.ts` | 수정 | loadSpecContent 교체 + loadSpecContentLegacy 보존 |
| `services/svc-skill/src/ai-ready/spec-content-adapter.test.ts` | **신설** | adapter unit test 4건 |
| `services/svc-skill/src/ai-ready/evaluator.test.ts` | 수정 | loadSpecContent 테스트 갱신 |

## 4. spec-content-adapter.ts 설계

### 입력: `SkillPackage` (from `@ai-foundry/types`)
```typescript
{
  skillId: string (UUID),
  metadata: { domain, subdomain?, language, version, ... },
  policies: Array<{ code, title, condition, criteria, outcome, source, trust, tags }>,
  trust: { level, score },
  provenance: { sourceDocumentIds, organizationId, extractedAt, pipeline },
  ontologyRef: { ... },
}
```

### 출력: `{ specContent: SpecContent; skillName: string }`

### SpecContent 매핑

| SpecContent 필드 | 소스 | 변환 방법 |
|-----------------|------|-----------|
| `originalRules` | `policies[]` | 전체 policies를 BL-style markdown 표로 변환 (1개 문서) |
| `emptySlotRules` | — | `[]` (skill.json에 ES-XXX 별도 없음) |
| `rules` | 위 둘 합집합 | `[...originalRules]` |
| `runbooks` | `policies[]` | 각 policy → runbook markdown 1건 |
| `tests` | `policies[]` | 각 policy → YAML 테스트 시나리오 1건 |
| `contractYaml` | `policies[]` + `metadata` | 전체 policy 코드 목록 + 버전 YAML |
| `provenanceYaml` | `provenance` + `trust` | 출처 + businessRules(policy codes) YAML |

### skillName 도출
```typescript
const skillName = [pkg.metadata.domain, pkg.metadata.subdomain]
  .filter(Boolean)
  .join("-")
  .toLowerCase()
  .replace(/\s+/g, "-");
```

## 5. evaluator.ts 변경 상세

### Before
```typescript
export async function loadSpecContent(env, skillId, organizationId): ...
  // reads spec-containers/{org}/{skillId}/manifest.json
```

### After
```typescript
// New (primary path)
export async function loadSpecContent(env, skillId, organizationId): ...
  // reads skill-packages/{skillId}.skill.json → adapter → SpecContent

// Legacy (preserved for Tier-A debugging)
export async function loadSpecContentLegacy(env, skillId, organizationId): ...
  // original spec-containers path — unchanged
```

**중요**: caller (routes/ai-ready.ts, queue/ai-ready-consumer.ts)는 수정 없음.

## 6. 테스트 계획

### spec-content-adapter.test.ts (4 unit tests)
1. `skillPackageToSpecContent` — policies → originalRules 표 포함 확인
2. `skillPackageToSpecContent` — provenanceYaml에 policy codes 포함 확인
3. `skillPackageToSpecContent` — runbooks 수 = policies 수 확인
4. `skillPackageToSpecContent` — tests 수 = policies 수 확인

### evaluator.test.ts (갱신)
- `loadSpecContent` — R2에 `skill-packages/{id}.skill.json` 있을 때 성공 반환
- `loadSpecContent` — R2에 없을 때 null 반환 (기존 동작 유지)
- `loadSpecContentLegacy` — manifest 없을 때 null 반환 (기존 테스트 이동)
