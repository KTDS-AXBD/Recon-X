## 3. Root Cause 진단 — TD-56

### 3.1 가설 정립

`packages/utils/src/llm-client.ts:118` `callLlmRouterWithMeta` 흐름:
1. `fetch(env.CLOUDFLARE_AI_GATEWAY_URL, POST, Authorization Bearer OPENROUTER_API_KEY)`
2. response.ok=true(HTTP 200) + body=HTML(`<!DOCTYPE`)이면 `response.json()`이 SyntaxError throw
3. `evaluator.ts:159` catch (e) → score=0, rationale=`Evaluation failed: ${String(e)}`

**Single evaluate** (Master smoke) ✅ 정상 → endpoint URL 자체 정상. **Batch Queue** ❌ 100% fail → Queue 환경에서만 systemic.

### 3.2 가설 후보

| # | 가설 | 가능성 | 검증 방법 |
|---|------|--------|----------|
| H1 | OpenRouter rate limit (free tier) — burst 시 일부 call HTML rate limit page | 중 | OpenRouter dashboard log + 단계적 throttle test |
| H2 | CF AI Gateway rate limit — 같은 burst 시 HTML index | 중 | CF AI Gateway dashboard 상태 + auth=False 모드 다시 점검 |
| H3 | CF Worker subrequest 한도 50/request 초과 — Queue handler에서 6 LLM call × 동시 batch_size 가능성 | 중상 | wrangler.toml ai-ready-queue `max_batch_size` 확인 |
| H4 | Anthropic API rate limit (OpenRouter 통한) | 중 | Anthropic dashboard 또는 OpenRouter passthrough 응답 헤더 |
| H5 | Queue context env vars drift — production env에서 staging URL 호출 | 낮 | Queue handler에서 env.CLOUDFLARE_AI_GATEWAY_URL 직접 logging |
| H6 | timing: Master smoke 직후 23s에 batch trigger → 캐시 invalidation race | 낮 | 5분 idle 후 batch 재시도 |

### 3.3 가설 시그널 강화

**H3 — CF Worker subrequest 한도 50/request 가설** 강화 시그널:
- batch 15초 fast-fail = 모든 LLM call이 시작과 거의 동시에 fail
- 8 skill × 6 LLM call = 48 subrequest 동시 시도 → 50 한도에 근접
- Queue concurrency 10 + 동시 처리 시 worker isolate 1개당 최대 60 subrequest 가능

**H1/H2 가설** — OpenRouter/CF AI Gateway free-tier rate limit:
- Master smoke 직후 burst 발생 시 token bucket 비어있음 → 즉시 limit
- 그러나 Master smoke 2차 retry (06:16:22, batch 후 ~10분)는 정상 → bucket 회복됨

**가장 likely**: H1+H3 조합 — OpenRouter burst rate limit + CF Worker subrequest budget 동시 압박. fix 방향:
- Queue concurrency 1~3으로 throttle (max_concurrency 10 → 3)
- evaluator에서 6 criteria를 sequential 호출로 변경 (현재 Promise.all 병렬)
- callLlmRouterWithMeta에 retry+exponential backoff (HTTP 200 + Content-Type !json 검출 시 재시도)

### 3.4 추가 진단 비용

정확한 root cause 확정에는 1~2h 추가 필요:
- `wrangler tail svc-skill --env production` 30초 + 동시 batch 재시도 → runtime error log 확보
- Queue concurrency 1 일시 변경 후 재배포 + batch 재시도 → burst 가설 검증
- packages/utils/src/llm-client.ts에 response.headers.get("content-type") logging 추가

본 Sprint 시간 효율 측면에서 진단 deferred. **TD-56 신규 등록**으로 별도 sprint 해소 권고.
