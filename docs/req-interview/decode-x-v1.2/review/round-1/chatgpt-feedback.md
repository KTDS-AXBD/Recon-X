## 2. 실행 가능성

    특히 핵심 서비스별 8~12개 Slot × Tier-A 다수 서비스에 대해 HITL 인터뷰, Runbook 역추출, 테스트 시나리오 작성 등은 정량적 추정보다 훨씬 많은 시간과 도메인 전문가 리소스가 필요할 가능성 큼.  
  - **프로토타입 → v1.1 스키마 및 Harness 전환 마이그레이션 리스크**:  
    문서 내에 상세한 4-Phase 마이그레이션 계획이 있음에도, 실제 운영 중인 조직·고객 환경에서 호환성 문제, 교육·적응 비용, 도구 안정성 문제 등 예상보다 큰 장애 발생 가능성 있음.  
  - **LLM 비용 및 모델 의존성**:  
    R1로 명시했으나, 대규모 Spec 생성, 반복 재생산성 검증 시 예상보다 훨씬 더 큰 비용 및 API 호출 지연, 모델 안정성 문제 발생 위험 상존.  
  - **Foundry-X와의 연동 및 버전 스큐 문제**:  
    R13에서 다루고 있으나, 양 팀 간 동기화 실패 시 타격 큼. 특히 Git 저장소·SPEC.md·Plumb 계약 버전 간 mismatch는 프로젝트 전체 지연 요인 될 수 있음.

## 3. 누락된 핵심 요소

- **운영 및 유지보수 계획 부재**  
  - Spec 산출과 Harness 운영 이후, Spec 및 Harness가 지속 진화·관리되는 프로세스와 조직 체계(특히 변경 관리, 버전 관리, 운영 중 Spec Drift 대응 프로세스)가 구체적으로 상세히 기술되어 있지 않음.  
  - 예: Spec 업데이트 시점, 도메인 전문가 재검증 절차, Spec 폐기/수정 시 영향도 분석과 롤백 정책 등.

- **보안 및 개인정보 보호 정책 구체화 부족**  
  - 프로젝트가 레거시 소스코드·운영 로그·도메인 민감 데이터를 다루는데, 민감 데이터 관리, 접근 권한 통제, 데이터 암호화, 감사 로그 등 보안 관련 구체적 정책과 기술적 대응책이 상세하지 않음.  
  - R14에서 민감 업무 노하우 유출 리스크를 다루긴 하나, 전체 보안 아키텍처 관점에서 더 구체적 설계 필요.

- **성공적인 도메인 전문가 참여 동기 및 교육 방안 미흡**  
  - HITL, Domain Archeologist 역할의 핵심 의존성에도 불구하고, 도메인 전문가의 참여 동기 부여, 교육, 인터뷰 표준화 방안, 품질 보증 방안이 부족함.

- **품질 보증 단계 내 자동화·모니터링 도구 부족**  
  - Differential Testing 결과에 대한 자동화된 리포팅, 알림, 대시보드 체계 등 실시간 모니터링·문제 추적 도구가 구체적으로 제시되지 않음.
